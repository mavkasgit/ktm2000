# FactoryPlan Specs

Этот каталог фиксирует продуктовые и технические решения для локальной системы планирования и контроля производства.

## Цель MVP

MVP должен проверить основной производственный контур на первом источнике данных: Excel-импорте. При этом Excel не является фундаментом системы. Фундаментом является общий производственный план, в который позиции могут попадать из разных источников.

Первый проверочный поток:

`Excel import -> production plan -> plan positions -> internal plan -> section tasks -> movements -> transfers -> final release`

Целевой поток:

`manual/API/integration/Excel source -> production plan -> plan positions -> execution`

## Главные решения v1

1. `ProductionPlan` и `PlanPosition` являются ядром планирования.
2. Excel остается первым источником позиций плана, но не единственным будущим способом наполнения плана.
3. Каждая позиция плана хранит источник: `manual`, `excel_import`, `api`, `integration`.
4. Внутренние primary keys: `bigint identity`, как в HRMS. Бизнес-номера (`plan_no`, `batch_no`, `transfer_no`) и source/dedup поля хранятся отдельно.
5. Общую техническую структуру берем из HRMS как референс: infra layout, FastAPI async foundation, axios/TanStack Query, Tailwind/Radix shared UI, Playwright layout.
6. `Movement` является журналом фактов и не редактируется после создания.
7. Запущенная позиция плана immutable по ключевым полям; изменения оформляются через `PlanAdjustment`.
8. План запускается в производство пакетами `ReleaseBatch`: ближайшие 2-3 дня, неделя, ручной запуск или предварительный future batch.
9. Маршрут фиксируется в момент запуска позиции. Если маршрут выбран неверно и факта еще нет, запуск отменяется и создается заново с правильным маршрутом.
10. Брак является отдельным объектом с решением: списать, переделать, вернуть на предыдущий участок, удержать на разборе или принять с отклонением.
11. Количественные итоги в задачах и планах считаются из журнала движений; кеши допустимы только как производные данные для быстрого отображения.
12. Следующий участок может работать только с количеством, которое принято от предыдущего участка.
13. Передача и приемка являются разными событиями, потому что между ними возможны расхождения.
14. Для MVP маршрут изделия линейный. Параллельные этапы, альтернативные маршруты и APS-оптимизация остаются за пределами v1.
