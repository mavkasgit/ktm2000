# 01. Domain Model

## Производственный поток

Базовый проверочный поток v1:

`ExcelImport -> ProductionPlan -> PlanPosition -> InternalPlan -> SectionPlanLine -> WorkTask -> Movement -> Transfer -> FinalRelease`

Целевая модель:

`PlanSource -> ProductionPlan -> PlanPosition -> execution`

`PlanSource` может быть `manual`, `excel_import`, `api` или `integration`. Excel нужен на первом этапе, но не должен становиться корнем доменной модели.

Логика похожа на связку ERPNext `Production Plan -> Work Order -> Job Card -> Stock Entry`, но в FactoryPlan акцент делается не на складских проводках, а на прозрачном движении количества между участками.

Ключевые решения:

- общий план является живым документом;
- запуск в производство выполняется пакетами `ReleaseBatch`, а не всем планом сразу;
- запущенная позиция плана immutable по ключевым полям;
- изменения запущенной позиции оформляются через `PlanAdjustment`;
- маршрут фиксируется в момент запуска;
- факт хранится в `Movement`, кеш итогов используется только для быстрого чтения.

## Сущности

### Product

Изделие, полуфабрикат, компонент или материал.

Типы:

- `finished_good`;
- `semi_finished`;
- `component`;
- `material`.

Правила:

- `sku` уникален;
- неактивное изделие нельзя использовать в новых планах;
- уже созданные исторические планы продолжают ссылаться на изделие.

### BOM

Состав изделия.

Для MVP BOM нужен как проверка технологической готовности изделия к планированию. Полное планирование материалов и нехваток не входит в v1.

Правила:

- у готовой продукции должен быть активный BOM;
- активный BOM только один на изделие;
- строки BOM не должны ссылаться на то же изделие напрямую;
- количество компонента должно быть больше 0.

### ProductionRoute

Маршрут производства изделия.

Для MVP маршрут линейный: `step 1 -> step 2 -> step N`.

Правила:

- у готовой продукции должен быть активный маршрут;
- активный маршрут только один на изделие;
- изменение маршрута создает новую версию, а не переписывает уже использованную;
- `sequence` внутри маршрута уникален;
- первый этап не требует приемки от предыдущего участка;
- последний этап может создавать `final_release`.

### Section

Производственный участок.

Правила:

- `code` уникален;
- участок может быть неактивным;
- неактивный участок нельзя использовать в новых маршрутах.

### ProductionPlan

Общий производственный план. Может иметь период, но не обязан быть строго месячным документом. Позиции могут добавляться постепенно из разных источников.

Статусы:

- `draft` - план создан, позиции можно добавлять и править;
- `validated` - нет блокирующих ошибок по активным позициям;
- `approved` - план или выбранный набор позиций утвержден;
- `partially_released` - часть позиций запущена в производство;
- `released` - все утвержденные позиции запущены в производство;
- `cancelled` - план отменен.

Переходы:

- `draft -> validated`;
- `validated -> approved`;
- `approved -> partially_released`;
- `approved|partially_released -> released`;
- `draft|validated|approved -> cancelled`.

После запуска позиции в производство нельзя менять ее количество, изделие, срок и маршрутный snapshot. Для изменения нужно создать новую позицию или корректировку.

### PlanPosition

Позиция производственного плана.

Ключевые поля:

- product;
- quantity;
- period_start / period_end;
- due_date;
- priority;
- source_type;
- source_ref;
- status.

Статусы:

- `draft`;
- `invalid`;
- `valid`;
- `approved`;
- `released`;
- `cancelled`.

Правила:

- позиция может быть создана вручную или импортом;
- позицию нельзя запустить без изделия, количества, BOM и маршрута;
- позиция после `released` становится immutable для плановых полей;
- повторный Excel-импорт должен ссылаться на источник, а не переписывать историю позиции без следа.

### PlanAdjustment

Корректировка запущенной или утвержденной позиции плана.

Типы:

- `increase_quantity`;
- `decrease_quantity`;
- `change_due_date`;
- `cancel_remaining`;
- `replace_position`.

Правила:

- корректировка не меняет исходную позицию задним числом;
- корректировка хранит причину, автора и дату;
- корректировка может создать новую позицию или уменьшить оставшуюся потребность;
- нельзя корректировкой сделать итоговую потребность меньше уже выпущенного или уже находящегося в необратимом факте количества.

### ReleaseBatch

Пакет запуска позиций плана в производство.

Типы:

- `near_term`;
- `weekly`;
- `future_preparation`;
- `manual`.

Правила:

- один общий план может иметь много пакетов запуска;
- пакет содержит выбранные позиции или части позиций;
- маршрут фиксируется на строке пакета запуска, чтобы разные частичные запуски могли иметь разные версии маршрута;
- задачи участков создаются из пакета запуска;
- отменить пакет можно только если по его задачам нет движений;
- если факт уже есть, отмена заменяется корректировкой.

### PlanChangeSet

Набор изменений производственного плана перед применением импорта или массовой правки.

Правила:

- пользователь видит diff до применения;
- примененный набор можно откатить, если затронутые позиции еще не были запущены в производство;
- запущенные позиции откатом не меняются, для них используется `PlanAdjustment`.

### InternalPlan

Производственная декомпозиция утвержденных позиций плана.

Статусы:

- `draft`;
- `active`;
- `completed`;
- `cancelled`.

### SectionPlanLine

Плановая строка участка для конкретного изделия и этапа маршрута.

Важно: поля итогов должны быть вычисляемыми из `Movement`. Если ради скорости они денормализованы, обновлять их можно только backend-сервисом в той же транзакции, где создается движение.

Для отображения маршрута система хранит кеш итогов по каждому этапу: план, доступно, выдано, выполнено, передано, принято, брак, остаток. Источник истины остается `Movement`.

### WorkTask

Задача участка.

Статусы:

- `draft`;
- `waiting_previous`;
- `ready`;
- `in_progress`;
- `partially_completed`;
- `completed`;
- `transferred`;
- `blocked`;
- `cancelled`.

Переходы:

- первый этап: `draft -> ready`;
- не первый этап: `draft -> waiting_previous`;
- при приемке количества: `waiting_previous -> ready`;
- при выдаче в работу: `ready -> in_progress`;
- при частичном выполнении: `in_progress -> partially_completed`;
- при полном выполнении: `in_progress|partially_completed -> completed`;
- при передаче всего годного количества: `completed|partially_completed -> transferred`;
- при ручной блокировке: любой активный статус -> `blocked`;
- при отмене: любой незавершенный статус -> `cancelled`.

Удаление задачи:

- физически удалить можно только задачу без движений в отмененном пакете запуска;
- если по задаче есть движения, задача не удаляется, а отменяется или закрывается корректировкой;
- если маршрут выбран неверно до начала факта, нужно отменить пакет запуска и создать новый с правильной версией маршрута.

### Movement

Неизменяемый журнал движения количества.

Типы:

- `issue_to_work`;
- `complete`;
- `transfer_send`;
- `transfer_receive`;
- `reject`;
- `scrap`;
- `return_to_previous`;
- `final_release`;
- `adjustment`.

`adjustment` разрешен только администратору и требует комментарий.

### Transfer

Документ передачи между участками. Нужен поверх `Movement`, потому что у передачи есть жизненный цикл и приемка.

Статусы:

- `draft`;
- `sent`;
- `accepted`;
- `partially_accepted`;
- `rejected`;
- `cancelled`.

Правила:

- `sent_quantity > 0`;
- принять больше отправленного нельзя;
- отклоненное количество равно `sent_quantity - accepted_quantity`, если приемка закрыта;
- приемка создает `transfer_receive`;
- расхождение создает `reject` или отдельное поле `discrepancy_quantity` с причиной.

### Defect

Брак или отклонение.

Статусы:

- `open`;
- `decision_required`;
- `rework_task_created`;
- `scrapped`;
- `returned`;
- `accepted_with_deviation`;
- `closed`.

Решения:

- `scrap`;
- `rework_current`;
- `return_previous`;
- `quality_hold`;
- `accept_with_deviation`.

Причины v1:

- `production_defect`;
- `material_defect`;
- `section_error`;
- `transfer_shortage`;
- `quality_mismatch`;
- `wrong_item`.

### ReworkTask

Отдельная задача на доработку брака.

Правила:

- создается из `Defect`, если выбрано решение `rework_current` или `return_previous`;
- связана с исходной задачей и дефектом;
- имеет собственные движения выполнения;
- результат доработки может вернуть годное количество в поток или списать остаток.

## Количественные инварианты

Для каждой `WorkTask`:

- `issued_quantity <= available_quantity`;
- `completed_good_quantity + rejected_quantity <= issued_quantity + accepted_rework_quantity`;
- `transferred_quantity <= completed_good_quantity`;
- `final_released_quantity <= completed_good_quantity` только для последнего этапа;
- количество в работе = `issued - completed_good - rejected - returned`;
- нельзя создать движение с `quantity <= 0`;
- нельзя удалить движение после создания.

Для передачи:

- `accepted_quantity <= sent_quantity`;
- `accepted_quantity + rejected_quantity <= sent_quantity`;
- следующая задача получает доступность только по `accepted_quantity`.

## Что считаем прогрессом

Прогресс позиции плана по изделию:

- `planned_quantity` - количество из позиции плана;
- `started_quantity` - количество, выданное в работу на первом этапе;
- `current_wip_quantity` - количество, не дошедшее до финального выпуска и не списанное в брак;
- `final_good_quantity` - количество с движением `final_release`;
- `scrapped_quantity` - списанный брак;
- `remaining_quantity = planned_quantity - final_good_quantity - scrapped_quantity`.

Процент выполнения:

`final_good_quantity / planned_quantity * 100`

Если нужно показывать техническую готовность по маршруту, это отдельная метрика:

`min(completed_good_by_step / planned_quantity)` по всем этапам.
