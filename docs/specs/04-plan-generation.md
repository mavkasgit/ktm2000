# 04. Plan Generation

## Цель

Из утвержденных позиций производственного плана создать пакет запуска, внутренний производственный план, строки планов участков и задачи участков.

## Вход

`ProductionPlan` с одной или несколькими `PlanPosition` в статусе `approved` и выбранный горизонт запуска.

Все строки должны иметь:

- `product_id`;
- `quantity > 0`;
- активный BOM;
- активный route;
- минимум один `route_step`.

## Выход

Для каждой выбранной `PlanPosition`:

- один `ReleaseBatch`;
- один `InternalPlan`, общий для партии запуска;
- `SectionPlanLine` на каждый этап маршрута;
- `WorkTask` на каждый `SectionPlanLine`.

## Алгоритм v1

1. Создать или открыть черновой `ReleaseBatch`.
2. Заблокировать выбранные `plan_positions` на время генерации.
3. Проверить, что суммарный запуск по позиции не превышает утвержденное количество.
4. Для каждой позиции плана найти активный маршрут.
5. Зафиксировать `route_id` и `route_version` на `ReleaseBatchPosition`.
6. Создать `InternalPlan(status='active')` для `ReleaseBatch`.
7. Отсортировать `route_steps` по `sequence`.
8. Для каждого этапа создать `SectionPlanLine`.
9. Для каждого этапа создать `WorkTask`.
10. Первому этапу поставить статус `ready`.
11. Остальным этапам поставить статус `waiting_previous`.
12. Инициализировать кеши количеств для задач и этапов.
13. Перевести выбранные `PlanPosition` в `released` или частично запущенное состояние, если запущена часть количества.
14. Если запущены не все утвержденные позиции плана, перевести `ProductionPlan` в `partially_released`.

## Quantity model

В MVP каждый этап получает `planned_quantity = plan_position.quantity`.

Это сознательное упрощение:

- BOM не взрывается в потребности компонентов;
- потери по нормативам не планируются заранее;
- полуфабрикаты не получают отдельные задачи, если они не являются отдельными позициями производственного плана.

Будущее расширение:

- `route_step.input_product_id`;
- `route_step.output_product_id`;
- коэффициенты выхода;
- плановые потери;
- параллельные этапы;
- альтернативные маршруты.

## Idempotency

Генерация должна быть идемпотентной:

- повторный вызов для уже запущенного `ReleaseBatch` возвращает существующий `InternalPlan`;
- повторный вызов по тем же позициям и количествам не создает дублей;
- при ошибке транзакция откатывается полностью.

## Отмена неверного запуска

Если маршрут или состав запуска выбран неверно:

- если по задачам `ReleaseBatch` нет движений, пакет можно отменить, задачи физически удалить или пометить `cancelled`;
- если движения уже есть, пакет не удаляется, а ошибка исправляется через `PlanAdjustment`, новую позицию или новый `ReleaseBatch`.

Это правило защищает историю производства.

## Due dates

В MVP:

- каждая задача наследует `due_date` из позиции плана;
- если `due_date` отсутствует, задача получает `null`;
- сортировка задач в UI использует `priority`, затем `due_date`, затем `created_at`.

Не делаем в v1:

- обратное планирование по нормативам;
- календарь рабочего времени;
- загрузку ресурсов;
- автоматическое расписание по сменам.

## Ошибки генерации

Генерация должна падать с понятной ошибкой, если:

- план не в статусе `approved`;
- строка невалидна;
- маршрут исчез или деактивирован после валидации;
- этап маршрута ссылается на неактивный участок;
- в маршруте нарушена последовательность;
- выбранное количество запуска превышает доступный остаток позиции.
