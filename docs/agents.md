## Сетевые режимы работы и реестр портов

Приложение поддерживает два параллельных режима доступа к портам:

1. **Режим общей сети (shared_gateway)**: Маршрутизация через единый reverse proxy на порту **80/443**.
2. **Автономный режим**: Прямой доступ по выделенным портам для каждого проекта.

| Порт | Назначение / Окружение | Автономный доступ | Сетевой доступ |
|---|---|---|---|
| **5180** | Frontend UI (dev) | Локально в dev | — |
| **8010** | Backend API (FastAPI) | dev / test / prod | — |
| **8081** | Nginx entrypoint (HRMS) | `http://localhost:8081` | `http://<domain>/` (порт 80) |
| **8082** | Nginx entrypoint (KTM-2000) | `http://localhost:8082` | `http://<domain>/` (порт 80) |
| **8083** | Nginx entrypoint (Экранчик) | `http://localhost:8083` | `http://<domain>/` (порт 80) |
| **9222** | Chrome CDP (отладка) | dev (тестирование) | — |

## Реестр субагентов и правила делегирования

Проект использует систему специализированных субагентов для автоматизации рутинных задач. Агенты вызываются через упоминание в чате: `@<имя_субагента> <инструкция>`.

### Список субагентов:

| Субагент | Вызов | Назначение |
|----------|-------|------------|
| `browser-checker` | `@browser-checker проверь страницу` | Выполняет действия в браузере: навигацию, снятие скриншотов, чтение логов консоли, сетевой анализ. Всегда делегируйте браузерные проверки ему, кроме случаев явной просьбы пользователя делать это напрямую. |
| `docs-maintainer` | `@docs-maintainer обнови документацию` | Отвечает за актуальность каталога `docs/`. Сканирует код на изменения и вносит соответствующие правки в md-файлы. Запускайте его после завершения задач. |
| `test-runner` | `@test-runner запусти тесты` | Автоматически запускает pytest или Playwright тесты и возвращает структурированный отчет. Только запускает и логирует, код не меняет. |
| `test-fixer` | `@test-fixer почини падающие тесты` | Запускает тесты, локализует ошибки в бэкенде или фронтенде, вносит исправления в код и проверяет успешное прохождение тестов. |
| `server-operator` | `@server-operator выполни shell` | Выполняет системные команды на сервере через SSH MCP сервер (управление docker, systemd, логами). Не предназначен для передачи файлов. |
| `sftp-operator` | `@sftp-operator загрузи файл` | Осуществляет передачу файлов (загрузка сборки, скачивание логов, бэкапов) на удаленный сервер через SFTP MCP сервер. Не выполняет системный шелл. |
| `server-deployer` | `@server-deployer разверни проект` | Координирует процесс деплоя: упаковывает локальные файлы, вызывает `sftp-operator` для заливки и `server-operator` для сборки и перезапуска контейнеров на сервере. |

## Ключевые правила разработки для агентов

1. **Использование CodeGraph:**
   Для поиска классов, методов, функций или связей в коде **запрещено** использовать массовый поиск `list_dir` или `grep_search`. Используйте специализированный MCP-сервер **CodeGraph** (команды: `callers`, `callees`, `query`, `impact`). Это экономит токены и ускоряет работу.
2. **Обработка Edge Cases (крайних случаев):**
   При изменении кода всегда учитывайте валидацию входных данных, отсутствие подключений к БД, обработку исключений и неверные форматы файлов (особенно при импорте Excel).
3. **Методология Feature-Sliced Design (FSD):**
   На фронтенде соблюдайте строгое разделение по слоям. Компоненты одной фичи не должны зависеть напрямую от другой фичи. Все зависимости должны идти сверху вниз: `app` -> `pages` -> `widgets` -> `features` -> `entities` -> `shared`.
4. **Кликабельные ссылки:**
   При предоставлении отчетов пользователю всегда делайте кликабельные ссылки на файлы с использованием схемы `file:///` (например, `[main.py](file:///c:/Users/user/VibeCoding/ktm2000/backend/app/main.py)`).
